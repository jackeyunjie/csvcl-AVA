## 一、实施策略

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **渐进增强** | 基于现有代码库逐步增强，每个阶段都有可用交付物 |
| **风险前置** | 核心基础设施（数据、通信、风控）优先建设 |
| **快速验证** | 每个阶段结束时有明确的验证标准和演示场景 |
| **1+1>2** | 充分利用现有MT5对接、数据处理、邮件通知等基础设施 |

### 1.2 现有资产盘点

| 现有组件 | 状态 | 复用方式 |
|----------|------|----------|
| MT5 ZeroMQ桥接 | 成熟 | 直接复用，增加断线重连 |
| State Hex引擎 | 成熟 | 直接复用，增加实时模式 |
| 收缩检测引擎 | 成熟 | 直接复用，增加多周期同步 |
| DuckDB数据库 | 成熟 | 扩展Schema，增加新表 |
| 信号评分系统 | 成熟 | 直接复用 |
| 邮件通知系统 | 成熟 | 直接复用，增加模板 |
| 定时调度器 | 成熟 | 直接复用 |
| CSV颜色标记 | 成熟 | 复用逻辑，Web化 |
| LLM分析器 | 可用 | 扩展为LLM网关 |
| 监控面板 | 基础 | 增强为Web界面 |

## 三、Phase 1: 基础设施强化 (第1-2月)

### 3.1 目标
建立统一的数据层、消息总线和Agent基础框架，实现观察Agent的MVP版本。

### 3.2 任务分解

#### Week 1-2: 数据层统一

**任务1.1: 新建qoder_state.duckdb数据库**
- 创建所有核心表（strategies, backtests, agent_messages, signals等）
- 设计索引和视图
- 编写初始化脚本
- **验收标准**: 数据库Schema完整，初始化脚本可一键执行

**任务1.2: 数据管道统一**
- 抽象DataSource接口
- 实现MT5DataSource和ParquetDataSource
- 统一数据质量检查
- **验收标准**: 所有数据源通过统一接口访问，数据质量自动检查

**任务1.3: 多周期数据对齐**
- 实现MultiTimeframeAligner
- 支持MN1/W1/D1/H4/H1对齐
- 验证对齐准确性
- **验收标准**: 多周期数据对齐误差<0.01%

#### Week 3-4: 消息总线建设

**任务1.4: 消息总线实现**
- 实现MessageBus（本地Queue模式）
- 定义标准消息格式
- 实现发布订阅机制
- **验收标准**: 消息延迟<10ms，支持1000消息/秒

**任务1.5: 事件定义**
- 定义所有系统事件类型
- 设计事件Payload结构
- 实现事件序列化
- **验收标准**: 所有事件类型定义完整，序列化/反序列化正确

#### Week 5-6: Agent基础框架

**任务1.6: Agent基类实现**
- 实现BaseAgent抽象类
- 实现生命周期管理
- 实现消息处理接口
- **验收标准**: Agent可注册、启动、停止、接收消息

**任务1.7: Agent编排器**
- 实现AgentOrchestrator
- Agent注册/注销管理
- 状态监控
- **验收标准**: 支持10个Agent同时运行，状态监控实时更新

#### Week 7-8: 观察Agent MVP

**任务1.8: 观察Agent开发**
- 实现收缩检测集成
- 实现突破检测逻辑
- 实现趋势跟踪状态机
- **验收标准**: 
  - 收缩检测准确率>80%
  - 突破检测延迟<5秒
  - 完整记录收缩→突破→趋势生命周期

| 里程碑 | 交付物 | 验证方式 |
