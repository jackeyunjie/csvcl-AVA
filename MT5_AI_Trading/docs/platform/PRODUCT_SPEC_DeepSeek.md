## 一、产品定位与愿景

### 1.1 一句话描述

**让交易者用自然语言描述交易想法, AI自动将其转化为量化策略, 通过多Agent辩论验证, 在多周期视角下完成从回测到执行的完整闭环。**

### 1.2 核心差异化

| 维度 | 传统量化平台 | TradingAgents类LLM平台 | **我们的平台** |
|------|-------------|----------------------|-------------|
| 策略来源 | 用户写代码/Pine Script | LLM分析基本面+新闻+舆情 | **LLM + 用户经验注入 + 自研因子引擎** |
| 验证方式 | 单次回测 | LLM Agent辩论 | **多Agent周期分层辩论 + Walk-Forward三阶段** |
| 周期视角 | 单一周期 | 单一周期 | **多周期共振: D1/W1(股票) + H1/M15(股指外汇)** |
| 收缩追踪 | 无 | 无 | **独家: BB+SR+ADX+Pivot四维收缩→突破→趋势跟踪全生命周期** |
| 执行对接 | 需单独部署券商API | Yahoo Finance只读 | **MT4/MT5 ZeroMQ原生对接, 已有基础设施** |
| 数据源 | 付费数据 | 免费Yahoo Finance | **MT5真实数据 + RDS MySQL持久化** |

### 1.3 目标用户画像

| 画像 | 技能 | 痛点 | 平台价值 |
|------|------|------|---------|
| **有经验的MT4/MT5交易者** | 会看K线, 有交易直觉但不写代码 | 想法无法量化验证 | 自然语言→策略, AI回测告诉你想法是否靠谱 |
| **量化学习者** | 会Python, 想进阶AI量化 | 不知道怎么把RL/LLM用到交易 | 提供完整案例: squeeze→突破→多Agent辩论 |
| **策略研究员** | 会写策略但效率低 | 手动调参太慢, 过拟合不自知 | 自动参数扫描+Walk-Forward+Agent交叉验证 |

## 三、F1: 自然语言策略生成

### 3.1 F1.1 对话式策略构建

**用户输入示例**:
```
"创建一个策略: M15周期上, 当布林带宽度低于20分位+ADX<12时检测为收缩setup,
然后用H1趋势方向过滤, 突破anchor后1bar确认入场。
做空止损80点, 盈亏比3:1, M15 K线走完确认离场。"
```

**系统处理流程**:

```
用户自然语言输入
  ↓
┌──────────────────────────────────────────────────────┐
│ Step 1: LLM意图解析 (DeepSeek/OpenAI/Claude)          │
│   提取: 周期=M15, 入场条件=BB+ADX收缩, 过滤=H1趋势    │
│   出场=止损80点+3:1, 品种=待用户选择                   │
├──────────────────────────────────────────────────────┤
│ Step 2: 策略骨架生成                                   │
│   映射内部API: squeeze detector, trend filter,         │
│   breakout detector, exit rules                       │
├──────────────────────────────────────────────────────┤
│ Step 3: 参数补全 (对模糊部分提问)                       │
│   "M15的ADX<12作为收缩阈值, 是否要调整?"               │
│   "anchor_range最小百分比用默认0.5%吗?"                 │
├──────────────────────────────────────────────────────┤
│ Step 4: 策略代码生成                                   │
│   生成Python策略类, 继承BaseStrategy, 实现:             │
│   - setup_detection()  收缩检测                        │
│   - trend_filter()     趋势过滤                        │
│   - breakout_check()   突破确认                        │
│   - exit_logic()       出场规则                        │
├──────────────────────────────────────────────────────┤
│ Step 5: 即时验证                                       │
│   自动获取最近30天数据, 跑一次快速回测                  │
│   返回: "该策略在最近30天中检测到3个信号, 2胜1负,       │
│          平均盈亏比2.8:1"                              │
└──────────────────────────────────────────────────────┘
```

### 3.2 F1.2 策略代码自动生成

**目标**: 支持生成4种策略类型:

| 策略类型 | 说明 | 示例输入 |
|----------|------|----------|
| **收缩突破型** (已有v5) | 基于BB/SR/ADX收缩+突破 | "M15收缩后做空" |
| **均线交叉型** | MA/EMA/加权均线系统 | "MA5上穿MA20买入" |
| **趋势跟踪型** | ADX+DI方向+移动止损 | "H1 ADX>25且+DI>-DI时做多" |
| **均值回归型** | 布林带/RSI超买超卖 | "价格触及BB下轨且RSI<30买入" |

**代码生成规范**:
- 生成的Python代码必须可直接导入运行
- 所有数值参数暴露为可调参数
- 含中文注释, 标记"此代码由AI生成, 请验证后使用"
- 策略输出通过标准接口: `signal(Timeframe, Symbol) -> {LONG, SHORT, HOLD}`

预设模板 (基于已有v5引擎验证过的参数):

### 3.4 F1.4 策略预览与编辑

- 左侧: 策略自然语言描述 + 参数编辑表单
- 中间: 生成的Python代码 (语法高亮, 可手动编辑)
- 右侧: 即时回测预览 (最近30天结果)
- 底部: "保存策略" "运行完整回测" "提交Agent辩论" 三个按钮

## 四、F2: 智能策略回测

### 4.1 F2.1 一键回测

**性能目标**: 30秒内返回结果 (单品种单周期, 365天数据)

**输入**:
- 策略 (代码或配置)
- 品种列表 (从14个白名单中选择)
- 数据周期 (M15/H1/H4/D1)
- 回测窗口 (默认365天, 可扩展到730天)

**输出**:
- 总setup数、唯一突破事件数、突破率
- 多周期共振分布 (顺势/逆势/中性)
- Walk-Forward三段: Train/Validation/Test
- 各类出场规则业绩对比
- 单个交易明细列表 (CSV可导出)

### 4.2 F2.2 多周期数据对齐

**核心机制**: `pd.merge_asof(direction='backward')` — 这是v3修正的重大Bug, 必须严格执行

```python
# 确保H4趋势判断使用的H4 bar是M15 bar时间戳之前最新已完成的
h4_data_aligned = pd.merge_asof(
    m15_data.sort_values('timestamp'),
    h4_data.sort_values('timestamp'),
    on='timestamp',
    direction='backward',  # 只使用 ≤ M15时间戳的H4 bar
    tolerance=pd.Timedelta('4h')
)
```

**验收标准**: 不能有任何未来函数泄漏, 通过未来函数检测脚本验证。

### 4.3 F2.3 Walk-Forward验证

```
全量数据
├── Train (60%): 最早期数据, 用于参数优化
├── Validation (20%): 中期数据, 用于超参选择
└── Test (20%): 最晚期数据, 用于最终评估 — 只看一次
```

**评估标准**:
- Test段净期望 > 0 → 策略通过
- Test段样本 ≥ 50 → 统计可信
- Test段与Train段期望差值 < 30% → 不过拟合

### 4.4 F2.4 出场规则对比

| 出场规则 | 说明 | 类型 |
|----------|------|------|
| `fixed_hold_5bar` | 入场后固定持有5根K线 | 规则型 |
| `structure_stop` | 突破前一个结构低点/高点止损 | 规则型 |
| `1r_partial_trail` | 盈利达1R时减仓50%, 剩余跟踪止损 | 动态型 |
| `1r_partial_stop` | 盈利达1R时减仓50%, 剩余初始止损 | 动态型 |
| `trailing_atr` | ATR动态跟踪止损 | 动态型 |

### 4.5 F2.5 参数敏感性分析

**扫描维度** (基于v5已验证的经验):

| 参数 | 默认值 | 扫描范围 | 步进 |
|------|--------|----------|------|
| max_adx | 12 | 10-14 | 1 |
| min_anchor_range_pct | 0.50 | 0.35-0.55 | 0.05 |
| cooldown_bars | 5 | 3-7 | 2 |

共 5×5×3 = 75组, 建议先跑27组(±1步进), 找到最优区域后再精确扫描。

## 五、F3: 多Agent协作系统

> 参考项目: **TradingAgents** (71K+ Stars, 分析师→研究员辩论→交易员→风控四层架构)、**CryptoTrader AI** (4Agent辩论+硬编码风控门)、**sudo-trade** (牛熊辩论+裁判)、**ContestTrade** (内部竞赛排名)

### 5.1 系统概述

我们采用**周期分层 + 角色分工**的混合Agent架构。与TradingAgents不同之处在于: 我们不模拟华尔街投研流程 (基本面+新闻+舆情+技术分析), 而是聚焦于**多周期视角下的量化策略验证**。

```
                         用户 (最终决策者)
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
        ┌──────────┐   ┌──────────┐   ┌──────────┐
        │ 研究Agent │   │ 执行Agent │   │ 风控Agent │
        │ (R-View)  │   │ (E-View)  │   │ (RK-View) │
        └──────────┘   └──────────┘   └──────────┘
              │               │               │
              └───────────────┼───────────────┘
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
        ┌──────────┐   ┌──────────┐   ┌──────────┐
        │ H1 Agent │   │M15 Agent │   │ D1 Agent │
        │ (已验证v5)│   │ (Phase1) │   │ (待构建) │
        └──────────┘   └──────────┘   └──────────┘
```

### 5.2 F3.3 研究Agent (R-View)

**职责**:
- 接收用户自然语言策略描述
- 调用策略生成引擎 → 转化为可执行策略代码
- 调用回测引擎 → 完成Walk-Forward验证
- 输出: 标准化的策略研究报告 (AgentReport格式)

**内部Agent细分**:

| 子Agent | 视角周期 | 资产类别 | 状态 |
|---------|----------|----------|------|
| R-H1-Squeeze | H1 @ H1/H4/D1 | 股指/外汇/贵金属 | ✅ v5完成 |
| R-M15-Squeeze | M15 @ H1/H4 | 股指/外汇 | ⏳ Phase1完成 |
| R-D1-Stocks | D1 @ W1 | 股票 | ⏳ 待构建 |
| R-State-Short | H1 @ State | 股指/外汇 | ⚠️ 已有数据待整合 |

**输出示例**:
```yaml
agent_id: "R-H1-Squeeze-v5"
recommendation: "推荐进入模拟盘"
confidence: "high"
metrics:
  net_win_rate: 0.685
  net_expectancy_pct: 0.271
walk_forward:
  test_expectancy_pct: 0.299
  test_sample_n: 58
```

### 5.4 F3.4 执行Agent (E-View)

**职责**:
- 管理模拟盘/实盘交易执行
- 每日扫描14品种的收缩状态
- 发现setup后按策略规则下单
- 记录所有交易用于绩效归因

**已有基础设施**:
- `run_v5_simulation.py` — 每小时扫描
- `observation_db.py` — 交易记录持久化
- `reification_agent.py` — 状态具体化

### 5.5 F3.5 风控Agent (RK-View)

**职责**:
- 仓位上限控制 (单品种不超过总资金的X%)
- 日内最大亏损熔断
- 连续亏损N笔后暂停交易
- 策略健康度监控 (衰减检测)
- 与CryptoTrader AI的硬编码风控门理念一致: 风控不依赖LLM, 用确定性的规则引擎

**风控检查清单** (每笔交易前):
```
[  ] 总仓位 < 30% 总资金
[  ] 单品种仓位 < 10% 总资金
[  ] 今日亏损 < 5% 总资金  
[  ] 连续亏损 < 5笔
[  ] 策略健康度 = GREEN (非 YELLOW/RED)
[  ] 不在新闻事件前后15分钟 (可配置)
```

### 5.6 F3.2 多Agent辩论引擎

**辩论流程** (受sudo-trade和TradingAgents启发):

```
用户: "我想用M15收缩突破做空DAX, H1趋势过滤, 止损80点"
  │
  ├─→ R-H1 Agent 分析:
  │     "H1视角下DAX net_win_rate=68.5%, 但这是H1的统计, 
  │      M15的胜率可能不同。建议Phase 2完整回测后再决定。"
  │
  ├─→ R-M15 Agent 分析:
  │     "M15已完成density扫描(18.7%), 共振25.2%, 
  │      但尚未做完整回测。我无法给出期望值。"
  │
  ├─→ R-State Agent 分析:
  │     "State策略short方向胜率92.8%, 支持做空DAX。
  │      但注意State策略和Squeeze策略的入场逻辑不同。"
  │
  └─→ 辩论合成器:
      ┌─────────────────────────────────────────────┐
      │ 共识:                                          │
      │  1. 三个Agent都支持DAX做空方向 (一致)            │
      │  2. H1 v5已验证正期望                           │
      │                                                │
      │ 分歧:                                          │
      │  1. M15未完整回测, 不能确认最佳参数              │
      │     → 需要用户决策: 先跑M15 Phase 2?           │
      │  2. State与Squeeze入场信号可能冲突               │
      │     → 建议两个策略独立运行, 不混合              │
      │                                                │
      │ 建议:                                          │
      │  H1 v5立即进入模拟盘 (高置信度)                  │
      │  M15等Phase 2回测后再加入 (中置信度)            │
      │  观察两周后对比两个Agent的实际表现               │
      └─────────────────────────────────────────────┘
```

**辩论规则** (继承AGENTS.md):
1. 每个Agent必须用同一口径输出 (净胜率、净期望、Walk-Forward三段)
2. Agent不能引用自己的假设, 只能引用回测数据
3. Agent之间可以互相质疑但必须用数据说话
4. 样本量不足的结论降权
5. 只输出统计可信的结论, 不可信的标记"需更多数据"

## 六、F4: 实时市场分析

### 6.1 F4.1 多周期结构共振分析

**监控维度**:

| 周期 | 监控内容 | 刷新频率 |
|------|----------|----------|
| MN1/W1 | 大趋势方向 (股票用) | 每周 |
| D1 | 支撑阻力位、关键枢轴 | 每日 |
| H4 | 中期趋势、收缩状态 | 每4小时 |
| H1 | 主交易周期、squeeze setup | 每小时 |
| M30 | 结构细化 | 每30分钟 |
| M15 | 精确入场、收缩突破检测 | 每15分钟 |

**共振判断**:
- H1 squeeze + H4 squeeze → 共振收缩, 高概率突破 ⬆️
- H1 squeeze + H4 trend → 收缩后顺势突破 ⬆️
- H1 squeeze + H4 none → 孤立信号, 降权 ⬇️

### 6.2 F4.2 收缩状态扫描

**已有实现**: `run_v5_simulation.py`

**扫描指标** (每15分钟刷新):
- squeeze_score (BB+SR+ADX, 满分5)
- anchor_range (收缩区间)
- trend_alignment (H4/D1趋势对齐)
- 是否满足setup条件

**输出**: 控制台表格 + CSV日志 + 日报汇总

### 6.3 F4.5 策略健康度监控

**三色灯系统**:

| 灯色 | 条件 | 动作 |
|------|------|------|
| 🟢 绿灯 | 模拟盘与回测偏差<20%, 样本充足 | 正常运行 |
| 🟡 黄灯 | 偏差20-50%, 或连续2周无信号 | 观察中, 准备参数调整 |
| 🔴 红灯 | 偏差>50%, 或连续4周负期望 | 暂停交易, 触发重优化 |
